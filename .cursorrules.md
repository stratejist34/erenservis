# Next.js Performance & Conversion Architecture Rules

> Bu kurallar her Next.js projesinde geçerlidir.
> Proje bazlı marka kimliği, renk paleti ve sektör detayları
> ayrı bir proje context dosyasında tanımlanır.

---

## 1. Architecture Philosophy

Static-first, conversion-aware, zero-waste architecture.
Her mimari karar şu üçlü filtreden geçer:

1. **Performans**: Bu karar Lighthouse skorunu düşürür mü?
2. **Dönüşüm**: Bu karar kullanıcıyı aksiyona yaklaştırır mı?
3. **Sürdürülebilirlik**: Bu karar 6 ay sonra teknik borç yaratır mı?

Bir karar bu üçünden birini bile riske atıyorsa, alternatif sun ve gerekçelendir.

---

## 2. Rendering & Data Strategy

- Default: Static Site Generation (SSG). Sayfa build-time'da üretilir.
- Server Side Rendering yalnızca runtime data veya authentication gerektiren sayfalarda.
- Build-time data fetching tercih edilir. API çağrıları mümkünse build sırasında yapılır.
- React Server Components varsayılandır. Client Components yalnızca interaktif UI için:
  form validasyonu, harita embed'leri, dinamik widget'lar, telefon tıklama takibi.
- Client Component kullanılacaksa gerekçesi açıkça belirtilir.
- Gereksiz hydration'dan kaçınılır. Statik UI asla client boundary içine alınmaz.

---

## 3. JavaScript Budget

- Next.js runtime overhead (~85KB gzipped) kabul edilir, bunun üstüne
  sayfa başına ek JS yükü 50KB'ı aşmamalıdır.
- Büyük kütüphaneler (Framer Motion, heavy map SDKs, date-fns full bundle)
  doğrudan import edilmez. Dynamic import veya hafif alternatif kullanılır.
- Üçüncü parti script'ler (analytics, chat widget) `next/script` ile
  `strategy="lazyOnload"` veya `afterInteractive` olarak yüklenir.
- Her yeni dependency eklenmeden önce bundle impact'i değerlendirilir.
  Eğer dependency sadece bir fonksiyon için gerekiyorsa, o fonksiyon elle yazılır.

---

## 4. Performance Targets

Hedefler Türkiye mobil ağı (4G, ortalama 30-50ms RTT) baz alınarak:

| Metrik | Hedef | Kabul Edilebilir Üst Sınır |
|--------|-------|---------------------------|
| TTFB | < 200ms (Cloudflare edge) | < 400ms |
| FCP | < 800ms | < 1.2s |
| LCP | < 1.5s | < 2.5s (Google "Good") |
| CLS | < 0.05 | < 0.1 |
| TBT | < 100ms | < 200ms |
| INP | < 100ms | < 200ms |

Eğer herhangi bir efekt, animasyon veya kütüphane bu hedefleri
riske atacaksa: UYAR, sebebini açıkla, daha hafif alternatif sun.

---

## 5. Image & Asset Strategy

- `next/image` zorunlu. Manuel crop/resize yapılmaz.
- Format: Build-time'da AVIF > WebP > JPEG fallback zinciri, `next/image` otomatik handle eder.
- `sizes` prop'u her image'da tanımlanır: mobil, tablet, desktop breakpoint'leri.
- Above-the-fold (hero, logo): `priority={true}`, preload.
- Below-the-fold: `loading="lazy"` (varsayılan).
- Dekoratif görseller mümkünse CSS ile (gradient, pattern, SVG inline) çözülür.
- Font dosyaları: `next/font` ile self-host, `display: swap`, subset: latin-ext (Türkçe karakterler).

---

## 6. Caching & Edge Strategy

- Cloudflare CDN katmanı Türkiye trafiği için zorunlu
  (Vercel edge Türkiye'de node yok, Istanbul Cloudflare POP'u var).
- Static asset'ler immutable cache header'ları ile servis edilir.
- HTML sayfaları edge'de cache'lenir, `stale-while-revalidate` stratejisi ile.
- API route'lar yalnızca zorunlu olduğunda oluşturulur;
  build-time data mümkünse API route kullanılmaz.

---

## 7. Streaming & Suspense

- React Suspense streaming yalnızca >200ms süren veri çağrıları için.
- Statik UI asla Suspense boundary içine sarılmaz.
- Skeleton/placeholder kullanılacaksa, CLS'yi artırmayacak şekilde
  gerçek içerikle aynı boyutta olmalı.

## 8. Scientific Conversion Architecture

### 8.1 Visual Hierarchy & Eye Flow

- Sayfa yapısı F-pattern (content-heavy) veya Z-pattern (landing/CTA) baz alır.
  Kullanılacak pattern, sayfanın amacına göre seçilir.
- Görsel hiyerarşi 3 katmanlıdır:
  1. **Primary**: Tek bir ana mesaj veya CTA — kullanıcının ilk 2 saniyede gördüğü şey.
  2. **Secondary**: Destekleyici bilgi — güven sinyalleri, fayda önermeleri.
  3. **Tertiary**: Detay bilgi — teknik açıklamalar, SSS, footer.
- Her sayfada yalnızca BİR primary CTA olur.
  İkinci bir CTA varsa görsel olarak daha düşük hiyerarşide kalır.

### 8.2 CTA Design Principles

- **Fitts Kanunu**: CTA butonları yeterli boyutta (min 48x48px touch target,
  ideal: tam genişlik mobilde), kolay erişilebilir konumda.
- **Isolation Effect (Von Restorff)**: CTA, çevresindeki elementlerden
  renk, boyut veya boşluk ile net şekilde ayrışır.
  CTA rengi sayfadaki diğer hiçbir elementte kullanılmaz.
- **Proximity (Gestalt)**: CTA, onu destekleyen bilgiye (fayda cümlesi,
  güven sinyali) yakın konumlanır. CTA asla bağlamsız, izole bırakılmaz.
- **Action-oriented copy**: "Gönder" veya "Tıklayın" değil,
  kullanıcının alacağı faydayı ifade eden metin:
  "Ücretsiz Arıza Tespiti İste", "Fiyat Teklifi Al", "Hemen Ara".
- **Urgency/scarcity yalnızca gerçekse**: Sahte sayaçlar veya
  yapay kıtlık sinyalleri kullanılmaz. Güven kırar.

### 8.3 Trust Signals

- Güven elementleri (müşteri yorumları, sertifikalar, garanti bilgisi,
  hizmet yılı, marka logoları) CTA'ya yakın konumlanır.
- Güven sinyalleri above-the-fold veya CTA'nın hemen üstünde/altında olmalı.
- Telefon numarası mobilde sticky header'da veya floating CTA olarak
  her zaman erişilebilir olmalı (click-to-call).

### 8.4 Cognitive Load Management

- Bir section'da 1 mesaj. Kullanıcıya aynı anda birden fazla karar sorulmaz.
- Form alanları minimize edilir. İlk temas için ad + telefon yeterli.
  Fazla alan = fazla sürtünme = düşük dönüşüm.
- Paragraflar kısa, taranabilir. Uzun metin blokları alt başlıklar ve
  görsel kırılımlarla bölünür.
- İkon ve görsel, metni destekler — dekorasyon için değil, anlam için kullanılır.

### 8.5 Mobile-First Conversion

- Türkiye'de trafik %75+ mobil. Tüm dönüşüm akışı önce mobilde tasarlanır.
- Mobilde CTA: thumb-zone'da (ekranın alt 1/3'ü), sticky veya floating.
- Mobilde telefon numarası tıklanabilir (`tel:` link), tek dokunuşla arama.
- Mobilde horizontal scroll, tiny butonlar, hover-dependent UI yasak.

## 9. Typography Hierarchy for Conversion

- Display font: Marka karakterini taşır. Başlıklarda kullanılır.
  Cesur, ayırt edici ama okunabilir olmalı.
- Body font: Temiz, yüksek x-height, geniş harf aralığı.
  Uzun metinlerde göz yormayan bir seçim.
- Font çifti uyumu: Kontrast yeterli ama kakofonik değil.
  Bir serif + bir sans-serif veya iki farklı ağırlıkta sans-serif.
- Tipografik hiyerarşi en az 3 seviye:
  H1 (hero/sayfa başlığı), H2 (section başlığı), body text.
  Her seviye arasında net boyut/ağırlık farkı olmalı (minimum 1.25x ratio).
- `next/font` ile self-host. Google Fonts harici istek yapmaktan kaçın.
  Subset: `latin-ext` (Türkçe ç, ğ, ı, ö, ş, ü karakterleri için zorunlu).

## 10. Anti-Patterns — Bunları Yapma

- Client Component'i "her ihtimale karşı" kullanma. Gerekçe yoksa Server Component.
- `useEffect` içinde veri çekme. Sunucu tarafında çöz.
- Layout shift yaratan lazy-loaded hero image. Hero `priority` olmalı.
- Tüm sayfayı tek bir Client Component'e sarma.
- CSS-in-JS runtime (styled-components, emotion) kullanma.
  Tailwind CSS veya CSS Modules tercih et.
- 100+ satır tek component dosyası. Bileşenleri böl, sorumlulukları ayır.
- Sahte urgency sinyalleri (fake countdown, "son 2 kişi bakıyor" gibi).
- Görsel efekt uğruna performanstan ödün vermek.
  Dekorasyon ikincil, hız ve dönüşüm birincil.

---

## 11. Decision Checkpoint

Her component veya sayfa oluşturmadan önce şu soruları geç:

```
[ ] Bu component Server Component olarak kalabilir mi?
[ ] JS bundle'a ne kadar ekleme yapıyor?
[ ] LCP'yi olumsuz etkiler mi?
[ ] Kullanıcıyı aksiyona (CTA) yaklaştırıyor mu?
[ ] Mobilde thumb-zone'da erişilebilir mi?
[ ] 6 ay sonra bakım sorunu yaratır mı?
[ ] Bu güven mi inşa ediyor, gürültü mü?
```

Bu kontrollerden geçemeyen bir element sayfada olmamalı.
